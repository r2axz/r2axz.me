Before I get to the point, I would like to share a short story to illustrate how
simple and obvious things sometimes get overlooked and lead to mistakes in
measurements and calculations.

A while ago, I asked a few fellow hams to solve a simple circuit problem shown on the schematics below.

![circuit problem schematics](/assets/2022-03-07-ltspice-net-statement/1.svg){: .post-image}
*Figure 1*

The question was, what would a calibrated VNA show on the S21 plot if we
connected an ideal 500-ohm resistor between its ports. Surprisingly, only a few
were able to come up with the correct answer. This was a little strange because
many of them used [NanoVNA](https://nanovna.com), and I expected some degree of
VNA fundamentals understanding. Anyway, let's see what went wrong.

First, let's make an equivalent circuit.

![VNA calibration](/assets/2022-03-07-ltspice-net-statement/2.svg){: .post-image}
*Figure 2*

Since the circuit in Figure 2 does not contain any frequency-dependent or
distributed components, a simple DC analysis will do the job. Let's apply
[KVL](https://en.wikipedia.org/wiki/Kirchhoff%27s_circuit_laws)

{% katex display %}
V - V_{R_S} - V_{R} - V_{R_L} = 0 \tag{1}
{% endkatex %}

{% katexmm %}
where $V$ is the voltage of the voltage source inside Port 1, $V_{R_S}$ is the voltage
drop on the Port 1 source impedance ($R_S$), $V_R$ is the voltage drop on the resistor R,
and $V_{R_L}$ is the voltage drop on the Port 2 load impedance ($R_L$).

Since [$V = IR$](https://en.wikipedia.org/wiki/Ohm%27s_law), Equation 1 can be rewritten as
{% endkatexmm %}

{% katex display %}
V - IR_S - IR - IR_L = 0 \Rightarrow\newline
\Rightarrow V = I(R_S + R + R_L) \tag{2}
{% endkatex %}

which gives

{% katex display %}
I = \frac{V}{R_S + R + R_L} \tag{3}
{% endkatex %}

To solve the problem, we must find the voltage on {% katex %}R_L{% endkatex %}

{% katex display %}
V_{R_L} = IR_L = \frac{R_LV}{R_S + R + R_L}\tag{4}
{% endkatex %}

Substituting values from the schematics into Equation 4 gives

{% katex display %}
V_{R_L} = \frac{1}{12}V \tag{5}
{% endkatex %}

Those who had succesfully arrived to this point would then most of the times
calculate S21 logmag value using the following math

{% katex display %}
S21\_Logmag = 20log_{10}(\frac{\frac{1}{12}V}{V}) \approx\newline\approx -21.58\ dB \tag{6}
{% endkatex %}

**Which unfortunately is wrong in the VNA context.**

{% katexmm %}
The reason it's wrong is that the S21 logmag plot 0 dB reference voltage is not
$V$ as it's implied by Equation 6.
{% endkatexmm %}

{% katexmm %}
During VNA calibration, the transmitting port (Port 1) is
connected to the receiving port (Port 2). Since both ports are matched, the
total power generated by Port 1 is equally distributed between Port 1 and Port 2
as the [maximum power transfer
theorem](https://en.wikipedia.org/wiki/Maximum_power_transfer_theorem) states.
This essentially means that the voltage on both ports is the same and equal to
$\frac{1}{2}V$.

Once calibration is done, $\frac{1}{2}V$ becomes the 0 dB reference voltage for
the transmission coefficient.
{% endkatexmm %}

*Note: the above is a huge oversimplification of what actually happens during
VNA calibration, but it's accurate enough for the needs of this post.*

So the correct form of equation for S21 logmag is

{% katex display %}
S21\_Logmag = 20log_{10}(\frac{\frac{1}{12}V}{\frac{1}{2}V}) =\newline
= 20log_{10}(\frac{1}{6})\approx -15.56\ dB \tag{7}
{% endkatex %}

{% katexmm %} -15.56 dB is the value you get if you measure a 500-ohm resistor
with a VNA or simulate the circuit in Figure 1 with
[RFSim99](https://www.electroschematics.com/rfsim99-download/) or other
VNA-aware software. However, if you perform AC analysis of the circuit in Figure
2 in
[LTspice](https://www.analog.com/en/design-center/design-tools-and-calculators/ltspice-simulator.html)
and plot the voltage across $R_L$, you will get -21.58 dB provided that your
voltage source amplitude is 1 V.

Let's go ahead and do that. I'm going to make two slight modifications to the
circuit in Figure 2. First, I will replace $R_S$ with the parasitic impedance of
the voltage source V. Second, I will add the  AC analysis statement
{% endkatexmm %}

```text
.ac list {1}
```

which tells LTspice to perform AC analysis using a single frequency of 1 Hz.

![LTspice AC analysis](/assets/2022-03-07-ltspice-net-statement/3.svg){: .post-image}
*Figure 3*
[Download this schematic](/assets/2022-03-07-ltspice-net-statement/ac_analysis.asc){: .schematic-download}

If we simulate the circuit and plot the decibel representation of VRL, we'll get
the following graph

![LTspice AC analysis plot](/assets/2022-03-07-ltspice-net-statement/4.svg){: .ltspice-plot}
*Figure 4*

which completely agrees with our predictions.

We need to remember that LTspice always uses 1 unit of whatever quantity you
measure as the 0 dB reference point for the decibel representation plot. In
other words, 1 V or A is 0 dB, 2 is 6.02 dB, etc., etc.

So the thing is, different software/hardware uses various dB references, and one
can (and almost certainly will) miss that when comparing measurements with
simulations. Fortunately, in the LTspice case, there is a way to make it behave pretty much
like a VNA by using the [.NET
statement](http://ltwiki.org/index.php?title=NET_Compute_Network_Parameters_in_a_AC_Analysis).

The .NET statement allows calculating a large number of things such as
S/Z/Y/H-parameters, admittance, and impedance of one or two-port networks and is
used in conjunction with AC analysis. Although the statement is fairly flexible
(see the above link for complete documentation), the recommended way of using it
is pretty simple. We need to have an independent voltage or current source (such
as V). Its parasitic series impedance Rser should be set to the desired system
impedance. We also need a load resistor used for termination or as a receiving
port. LTspice will then measure the current via the termination resistor, do its
internal trickery and generate new plot traces as if we were using a fairly
advanced network analyzer.

Let's modify the schematic in Figure 3 and add a .NET statement.

![LTspice AC analysis](/assets/2022-03-07-ltspice-net-statement/5.svg){: .post-image}
*Figure 5*
[Download this schematic](/assets/2022-03-07-ltspice-net-statement/net_analysis.asc){: .schematic-download}

If we run the simulation, switch to the plot window, and choose Add Traces
(Ctrl/Command-A), we'll see a bunch of new traces that were unavailable
previously. Let's plot S21(v) and V(vrl).

![LTspice AC analysis plot](/assets/2022-03-07-ltspice-net-statement/6.svg){: .ltspice-plot}
*Figure 6*

Finally, the green trace (S21) agrees with our calculations in Equation 7.

Personally, I believe the .NET statement is a huge deal because it allows me to
create models of circuits that I usually measure on my VNA and get consistent
results. But the .NET statement itself goes far beyond that, and there's
definitely a lot to explore in the future.
